Rust + WebAssembly 学習アプリ

WebAssembly は、ブラウザ上で動くバイナリフォーマットです。 Rust コードをコンパイルすると .wasm ファイルになり、 JavaScript から高速な関数として呼び出せます。 以下の 3 パネルで、そのしくみを実際に体験してください。

なぜ Wasm が速いのか？

JavaScript は動的型付き言語です。変数の型は実行時に決まり、 エンジンは「この変数は数値か？文字列か？」を毎回確認します。 WebAssembly は静的型付きのスタックマシンです。 すべての型はコンパイル時に確定しており、型チェックは不要です。

Wasm の関数はコンパイル後の .wasm 内でこう見えます：

;; export セクション: JS から呼べる関数の一覧
(export "fibonacci" (func $fibonacci))

;; type セクション: すべての型はコンパイル時に確定
(type (func (param i32) (result i64)))

;; Wasm の型は 4 種類だけ: i32, i64, f32, f64
;; 動的型付けなし → 型チェックのオーバーヘッドなし

Panel 2: Linear Memory

ゼロコピー共有

WebAssembly のLinear Memoryは、Wasm と JS が共有する 単一のフラットなバイト配列です。Rust がそこにデータを書き込み、 JS が同じメモリを直接読み取ります。コピーは発生しません。

「Wasm で書き込む」を押すと Rust がバッファに直接書き込みます。
ゼロコピーとは？

JS で new Uint32Array(len) を作ると、そのバッファは Wasm の Linear Memory の一部になります。 fill_fibonacci_sequence(buffer) を呼ぶと、 wasm-bindgen は「バッファのポインタと長さ」を Wasm に渡します。 Rust 側では &mut [u32] としてその領域を直接操作します。 JS はその後、同じ Uint32Array を読むだけで結果を得ます。 

;; memory セクション: Linear Memory の宣言
(memory (export "memory") 1)
;; 1 ページ = 64 KiB (65,536 バイト)
;; JS から: wasmModule.memory.buffer で ArrayBuffer として参照可能
;; Rust から: Vec, slice などが使うヒープ領域

JS コード側： 

const buffer = new Uint32Array(16);
fill_fibonacci_sequence(buffer);  // Rust が直接書き込む
console.log(buffer[7]);           // コピーなしで読める

// Linear Memory の実体を確認
const memory = wasmModule.memory;
console.log(memory.buffer.byteLength);  // 例: 65536 バイト

Panel 3: モジュール構造（WAT）

バイナリ構造

.wasm バイナリは決まったセクション構造を持ちます。 各カードをクリックして、セクションの役割と実際の WAT（テキスト形式）を確認してください。 WAT は wasm2wat fib_wasm_bg.wasm コマンドで生成できます。

§1
Type セクション
関数シグネチャ（型）の定義
;; すべての関数型をここで定義する
;; Wasm は静的型付き → 型はコンパイル時に確定
(type $t0 (func (param i32) (result i64)))
;; fibonacci: i32 を受け取り i64 を返す

(type $t1 (func (param i32 i32)))
;; fill_fibonacci_sequence: (pointer: i32, length: i32)
;; スライス = ポインタ + 長さ の 2 つの i32

;; Wasm の基本型は 4 つだけ:
;;   i32 (32bit整数), i64 (64bit整数)
;;   f32 (32bit浮動小数), f64 (64bit浮動小数)
§2
Import セクション
JS ホストから借りる機能
;; Wasm はサンドボックス: I/O は自力でできない
;; JS ホストから必要な機能を "import" する

;; console.log を JS から借りる
(import "wbg" "__wbg_log_..."
  (func $log (param i32 i32)))

;; メモリアロケーター (Rust の Vec 等のため)
(import "wbg" "__wbindgen_malloc"
  (func $malloc (param i32 i32) (result i32)))

;; Rust の extern "C" { fn log(s: &str); } がこれに対応
;; JS = ホスト, Wasm = ゲスト。ゲストがホストに頼む。
§5
Memory セクション
Linear Memory の宣言
;; 1 ページ = 64 KiB (65,536 バイト)
(memory (export "memory") 1)

;; Linear Memory の特徴:
;;   - アドレス 0 から始まる連続バイト配列
;;   - Rust の Vec<T> や String はここに配置される
;;   - JS から WebAssembly.Memory.buffer で参照可能
;;   - memory.grow(n) で 64KiB × n ページ分拡張できる
;;   - OS のスタック/ヒープとは別物

;; Panel 2 で確認したゼロコピーはこのメモリを使っている
§7
Export セクション
JS に公開する関数・メモリ
;; JS が呼べるのは export された関数だけ
;; #[wasm_bindgen] を付けた pub fn が export される

(export "fibonacci" (func $fibonacci))
(export "fill_fibonacci_sequence"
  (func $fill_fibonacci_sequence))
(export "get_module_info" (func $get_module_info))
(export "memory" (memory 0))

;; fib_inner は export なし → JS からは見えない
;; Wasm でも情報隠蔽 (pub/private) が機能する
§10
Code セクション
関数の実際のバイトコード
;; fibonacci 関数の WAT 表現（簡略化）
;; Wasm はスタックマシン: 値をスタックに積み、操作して返す
(func $fibonacci (param $n i32) (result i64)
  (call $fib_inner  ;; fib_inner を呼ぶ
    (local.get $n)))  ;; 引数をスタックに積む

(func $fib_inner (param $n i32) (result i64)
  ;; n <= 1 なら n を i64 に変換して返す
  (if (result i64)
      (i32.le_u (local.get $n) (i32.const 1))
    (then (i64.extend_i32_u (local.get $n)))
    (else
      (i64.add
        (call $fib_inner (i32.sub (local.get $n) (i32.const 1)))
        (call $fib_inner (i32.sub (local.get $n) (i32.const 2)))))))

実際の WAT を見るには： sudo apt install wabt && wasm2wat www/pkg/fib_wasm_bg.wasm を実行してください。また www/pkg/fib_wasm.js を読むと wasm-bindgen が生成した型変換グルーコードを確認できます。
